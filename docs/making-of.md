# Making of God Workbench

这份文档记录 God Workbench 的制作过程、关键判断、踩坑修正和可复用方法。

它不是产品说明，也不是部署手册。它回答的是：

- 这个产品为什么会长成现在这样
- 哪些判断改变了产品方向
- 哪些坑以后不应该再踩
- 这套制作方式如何迁移到其他产品

## 为什么做这个产品

God Workbench 起点不是“做一个游戏平台”，而是解决主持人真实流程里的后台负担。

“国王与天使”的趣味主要发生在 QQ 群和微信群里：匿名完成愿望、互相猜身份、试探、起哄、误判和揭晓。产品如果把这些部分完整搬进一个平台，反而会削弱游戏原有的人情味。

真正值得产品化的是上帝背后的重复劳动：

- 每轮维护成员名单
- 发布主题和截止时间
- 收集并录入愿望
- 按规则隐藏当前天使自己的愿望
- 防止重复选择、自选、漏选
- 处理最后交换和人工分配
- 生成揭晓文案、表格、图片和归档
- 保存当前进度，避免手工备份

所以这个产品的核心定位是主持人工作台，不是完整游戏平台。

## 产品是怎么长出来的

### 1. 从静态工作台开始

第一版先做成轻量网页工具，把主持人需要做的事情集中到一个页面里。

这一步的重点不是技术完整，而是把真实流程变成可操作界面：

- 成员名单
- 本轮主题
- 愿望录入
- 盲选推进
- 完成观察
- 揭晓生成
- 归档

这证明了产品方向：它可以减少 Excel 和手工记忆带来的错误。

### 2. 从手动导入导出转向自动保存

最初补了导入导出和备份能力，但很快暴露出一个判断问题：

如果日常保存依赖用户手动导入导出，产品实际上把复杂度推给了用户。

更合理的路线是：

- 所有操作自动保存
- 用户打开页面就继续上次状态
- 导入导出只作为逃生备份
- 日常不要求用户理解数据文件

这条判断后来影响了本地存储、云端同步和桌面快捷启动。

### 3. 接入 Supabase，但不强制登录

后续目标不是短期本地、长期云端两套路，而是直接按最终路线思考：数据应该保存到 Supabase。

但产品入口不能强制登录，因为页面内容和游戏流程本身是公开的，登录只是为了解决跨设备保存和长期保存。

因此当前原则是：

- 未登录也能进入页面并使用产品
- 未登录时自动保存到当前浏览器本地
- 登录后自动同步到 Glimmer Supabase
- 登录不是成员账号体系，也不是产品入口门槛

### 4. 明确成员名单不是顺序表

一次重要修正来自成员名单添加。

成员名单本质是参与者池，表示“哪些人可以参加游戏”。它不是本轮流程队列，不决定谁先许愿、谁先选愿望，也不决定最后如何交换。

真正有业务含义的顺序，是愿望提交顺序：

- 谁先把愿望发给上帝，谁就先被录入
- 盲选顺序按愿望提交顺序走
- 截图发送顺序按愿望提交顺序走
- 最后两人交换判断也基于愿望提交顺序

这条规则已经写入产品文档和协作规则，避免后续再把成员名单误当作排序表。

### 5. 增加本轮上帝选择

如果没有成员，第一次进入产品应该先添加成员，上帝自己也要在成员名单里，因为上帝由成员之一轮流担任。

如果已经有成员，开一轮游戏的第一步应该是选择本轮上帝：

> 本轮的上帝，你是谁？

这个动作不是登录，也不是账号体系，只是本轮主持身份选择。它让后续主题文案、录入、揭晓都能围绕本轮主持人展开。

### 6. 桌面快捷启动服务真实使用

产品不只需要能跑，还要降低打开成本。

桌面快捷启动的意义是让上帝像打开普通应用一样打开工作台，而不是每次记命令、找端口、配置环境变量。

快捷启动目前承担：

- 启动本地 Vite 服务
- 注入 Supabase 环境变量
- 复用已有服务
- 避免端口冲突
- 打开固定本地地址

这属于产品体验的一部分，不只是开发便利。

## 关键产品判断

### 产品服务主持人，不接管群聊

产品负责降低主持人后台错误，不负责替代 QQ 匿名完成和微信群猜身份。

如果某个功能会把游戏变成任务管理系统，应该先质疑。

### 自动保存优先于导入导出

导入导出适合作为逃生备份，不适合作为日常保存方式。

日常数据保存应该自动发生，用户不应该为了正常使用去理解文件、备份、恢复和版本。

### 登录是保存能力，不是入口门槛

这个产品的大部分内容不需要权限保护。强制登录会提高进入成本。

登录只用于云端自动保存和跨设备继续，不应该变成成员账号体系。

### 成员名单是池，愿望顺序才是队列

成员名单只表示参与者全集。

本轮业务顺序从愿望提交开始产生。

### 文档不是补丁，是产品资产

这个项目的文档分工应该清楚：

- `AGENTS.md`：协作规则和长期执行约束
- `docs/product.md`：产品定位、边界和流程
- `docs/progress.md`：当前完成度、欠缺和未来方向
- `docs/deployment.md`：部署和环境配置
- `docs/making-of.md`：制作过程、关键判断和可复用方法

## 踩坑与修正

### 添加名单被误解成顺序保存

问题：

成员名单添加曾被误解为“按这份名单顺序保存”，甚至接近覆盖或重排的语义。

修正：

成员名单添加应该是追加缺失成员，跳过重复成员。除非用户明确要求，否则不应该替换、重排或删除已有成员。

沉淀规则：

涉及数据变化时，执行前必须说清楚这次操作属于追加、替换、重排、删除、迁移还是只读。

### 执行前没有先解释操作方法

问题：

看似简单的任务如果直接执行，容易把用户意图误解成另一种数据操作。

修正：

未来执行前要先用用户能听懂的方式说明：

- 我理解你要解决什么
- 我会怎么做
- 会不会改已有数据
- 会不会影响顺序
- 会不会删除或覆盖

有歧义时先确认，再执行。

### 不应该直接把浏览器存储当操作入口

问题：

如果为了快，直接改 localStorage 或数据库，会绕开产品自身逻辑，隐藏真实问题。

修正：

优先通过产品界面或已有产品脚本操作。只有在修复损坏数据、迁移或排障时，才考虑直接操作底层存储，并且要提前说明风险。

### 文档要固定规则，而不是只靠记忆

问题：

口头修正无法保证后续协作一致。

修正：

凡是影响长期协作、数据语义、产品边界和执行方式的规则，都应该写入 `AGENTS.md` 或项目文档。

### 真实使用问题要进入浏览器级测试

问题：

很多体验问题不是模型层测试能发现的，例如按钮点击后页面自动下滑、窄屏表格压缩导致文本框暴涨、表头换行、字号失控、完成一轮后误改上帝再改回等。

修正：

把测试分成两层：

- `npm run test:regression`：保护业务规则和状态语义。
- `npm run test:e2e` / `npm run test:visual`：保护真实页面操作和渲染体验。

沉淀方法：

当用户连续通过截图发现 UI/流程问题时，不应该只修当前截图，而要把对应风险变成可重复的浏览器测试。浏览器测试必须使用沙盒上下文或种子数据，不能污染用户真实本地存档。

### UI 残留不能只修截图里的单点

问题：

选上帝后出现“发主题”跳转按钮，表面上像是一个按钮没删干净。但真正的问题不是这个按钮本身，而是旧页面逻辑是否还留下了一整组残留：旧渲染函数、旧 CSS、旧媒体查询、旧动效、旧测试命名。

如果只删截图里的按钮，产品短期看起来好了，但旧流程的样式和概念还会藏在代码里，后面改布局、改响应式或改交互时又可能重新冒出来。

修正：

排查残留时要从“这个元素属于哪条旧流程”开始，而不是从“截图里这个点怎么删”开始。

应该同时检查：

- 页面渲染函数和事件处理
- CSS 选择器、hover、transition、media query
- 测试断言和测试描述
- 构建产物里是否还包含旧选择器
- 真实页面 DOM 是否还渲染旧元素

沉淀规则：

关键词扫描只能提供线索，不能直接决定删除。每个命中都要回到当前渲染路径判断：有些旧名字是真残留，有些名字仍然服务当前产品概念。

这次保留了“旧跳转按钮不应存在”的测试断言，因为这种测试虽然提到旧 class，但作用是防止残留 UI 回归。

## 可复用方法

### 1. 先还原真实人工流程

不要从功能列表开始。

先问：

- 人现在是怎么完成这件事的
- 哪一步最费脑
- 哪一步最容易错
- 哪一步必须保留人的自由判断
- 哪一步可以被系统托住

God Workbench 的关键不是“做愿望系统”，而是看清上帝真实工作流。

### 2. 区分产品化和保留原状

不是所有流程都应该放进产品。

适合产品化的部分通常是：

- 重复
- 容易漏
- 有规则
- 需要格式化输出
- 需要保存历史

应该保留原状的部分通常是：

- 社交氛围
- 玩笑和试探
- 人的判断
- 不稳定但有趣的互动

### 3. 让系统吸收复杂度

如果一个功能要求用户记很多步骤、手动备份、手动恢复、手动同步，那只是把复杂度换了个地方。

更好的方向是：

- 自动保存
- 自动恢复
- 自动生成文案
- 自动隐藏不该看到的信息
- 自动识别异常场景
- 只在需要人判断时停下来

### 4. 把名词语义讲清楚

产品里的普通词会携带规则。

例如：

- 成员名单：参与者池
- 愿望顺序：本轮业务顺序
- 上帝：本轮主持人
- 登录：云端保存身份
- 归档：历史轮次快照

一旦名词语义不清，界面、代码和协作都会出错。

### 5. 先质疑，再执行

执行不应该等于照字面做。

尤其是涉及长期规则、数据、文档、部署和产品边界时，要先问：

- 这个请求背后的真实问题是什么
- 有没有更直接的路径
- 如果从零设计，会不会这样做
- 这会不会增加用户负担
- 这会不会破坏已有产品边界

质疑不是拖慢执行，而是避免把错误做实。

## 2026-06-13 完成状态收窄

### 当时的问题

完成状况记录用了未观察到、可能完成、已完成、需提醒四种状态，但上帝实际只需要知道愿望有没有完成。

### 做出的判断

完成状态不应该模拟观察过程。刚盲选结束时默认都是未完成；只有明确知道完成时才标记为已完成。不确定和待提醒在产品语义上都等同于未完成，群提醒通过统一复制文案解决。

### 实际修改

完成状态改成未完成、已完成二态；旧状态兼容折叠为未完成；完成记录表改成更高密度的三到四栏布局。

### 验证方式

用单元测试覆盖旧状态兼容和界面文案，并运行项目检查。

### 可复用经验

状态数量要来自真实决策，而不是来自“可能有这种情况”的想象。工具型产品里，状态越多，用户越像在维护系统，而不是完成任务。

## 当前回顾结论

God Workbench 目前最有价值的方向不是扩大成游戏平台，而是继续成为一个顺手、可靠、低负担的主持人驾驶舱。

后续优化应该优先围绕：

- 打开即继续
- 数据自动保存
- 截图和复制顺手
- 盲选不出错
- 揭晓好看
- 文档能让未来继续迭代

这个产品的制作经验也可以迁移到其他小工具：先守住真实流程，再把重复、易错、格式化和保存交给系统。
