# Product Progress

## 当前阶段

God Workbench 现在是一个可本地使用、可云端同步的主持人工作台。

产品入口不要求登录。未登录时，数据自动保存到当前浏览器本地；登录后，数据自动同步到 Glimmer Supabase 项目。登录只服务于跨设备保存，不是进入产品的门槛。

## 已完成

- 成员名单：跨轮复用，支持批量粘贴添加；语义是参与者池，添加时应追加缺失成员并跳过重复成员，不承担游戏流程顺序。
- 本轮上帝：已有成员名单时，开轮第一步从成员名单中选择本轮上帝；开新一轮会重新选择。
- 本轮设置：轮次、上帝、主题、定题日、许愿/完成/揭晓时间。
- 主题文案：生成可复制的微信群主题发布文案，目标体验是右侧实时预览、一键复制到群聊。
- 愿望录入：未许愿成员排在上方，已许愿成员排在下方并按许愿顺序编号；支持更新、删除和拖拽调整已许愿顺序；已提交但未分配的愿望显示“待选择”。
- 催愿望：支持编辑单人催愿望模板，并在每个未许愿成员行右侧复制对应文案。
- 盲选截图：至少两人提交愿望后即可先开始；按许愿提交顺序给当前天使展示可选愿望，隐藏自己的愿望、已被选走的愿望，并禁用会导致后续分配死局的选择。
- 盲选推进：上帝记录当前天使选中的愿望后，系统推进到下一个按许愿提交顺序排到的成员；后续补交愿望会追加到队尾；支持撤回和重置。
- 盲选恢复：即使选择队列走到末尾，仍保留撤回、重置；不在主界面提供复杂的人工分配面板。
- 最后异常处理：强制交换保留为旧数据兜底；正常盲选应提前禁用会造成最后死局的选择。
- 完成状况记录：支持未完成、已完成二态；旧的未观察到、可能完成、需提醒会按未完成兼容显示。
- 揭晓生成：支持揭晓文案、Markdown、Excel TSV、CSV、SVG 图片导出。
- 归档：支持归档本轮、更新同轮归档、恢复归档轮次。
- 自动保存：所有操作自动写入本地存储。
- 云端同步：可选 Supabase 登录后自动拉取和保存当前工作台状态。
- 逃生备份：支持下载完整 JSON 备份，不作为日常保存方式。
- 桌面快捷启动：桌面 `上帝工作台.app` 会启动本项目、注入 Supabase 配置、复用已有服务。
- 快捷启动检查：`npm run dev` 前检查 Node 版本、依赖和端口。
- 部署文档：已记录腾讯云静态托管和 Supabase 环境变量。
- 异常回归集：已建立 `npm run test:regression`，覆盖完整轮次误操作、成员变动、愿望变动、盲选恢复、旧数据兼容、归档恢复等核心状态异常。
- 浏览器级 P0/P1 检查：已建立 Playwright 测试，`npm run test:e2e` 覆盖真实主持流程和高风险误操作，`npm run test:visual` 覆盖字号、窄屏溢出、表头、顶栏、长愿望等体验烟测，`npm run test:browser` 一次跑完。

## 当前产品边界

应该继续服务主持人，不把群聊玩法搬进产品。

产品负责：

- 减少 Excel/手工处理错误
- 管理主持人需要掌握的信息
- 生成可复制、可截图、可导出的交付物
- 保存和同步主持人的工作台状态

产品顺序规则：

- 成员名单顺序只服务于查找和管理，不决定游戏流程。
- 愿望提交顺序才是本轮业务顺序。
- 愿望表里的“许愿顺序”只给已许愿成员生成；拖拽已许愿行会重排愿望提交顺序。
- 盲选顺序、截图发送顺序和最后交换判断都应来自愿望提交顺序；后补愿望继续追加到这个顺序里。

产品不负责：

- QQ 匿名完成过程
- 微信群猜身份过程
- 成员打卡
- 群聊氛围、玩笑、试探和自由互动
- 面向所有成员的完整游戏平台

## 欠缺和风险

- 云端冲突处理还很简单：同一账号多设备同时编辑时，后保存的状态会覆盖先保存的状态。
- 登录体验很基础：只有邮箱/密码登录和注册，没有找回密码、邮箱确认提示细节、OAuth。
- 云端同步没有显式“最后同步时间”和失败重试队列。
- 本地自动快照还没有做：清空、开新一轮、恢复归档前没有可视化回滚记录。
- 数据结构目前是整包 JSON 保存，不是细粒度表结构；这适合当前主持人工具，但不适合多人实时协作。
- 桌面快捷启动依赖本机项目路径 `/Users/yuchao/Documents/GitHub/god-workbench`，换机器后需要重编译或改脚本。
- 浏览器测试目前覆盖 Chromium；后续如果要面向更多真实环境，还需要补 Safari/WebKit、Firefox 或移动真机检查。
- Glimmer Supabase 项目里存在一些既有旧表/函数安全 advisor 告警，不是当前 `god_workbench_states` 新表引入，但后续应该单独清理。

## 下一步优先级

1. 云同步可见性

显示最后同步时间、同步失败原因、重试状态。用户不用懂 Supabase，但要知道数据是否安全。

2. 本地恢复记录

在清空、开新一轮、恢复归档、云端覆盖本地前自动生成快照，并提供恢复入口。

3. 表格截图友好

让盲选表格天然适合手动截图：第一列花名容易避开，愿望列清楚，控制项不干扰截图区域。

4. 揭晓图片优化

把 SVG 揭晓图做得更适合微信群分享，解决长愿望截断和多人轮次排版问题。

5. 部署到腾讯云

配置静态托管环境变量，确认 Supabase Auth redirect URL、邮箱注册流程和生产域名。

6. 云端数据模型评估

如果未来要做成员提交链接或多人协作，再从整包 JSON 转向结构化表。不要提前拆表，除非产品边界真的扩大。

## 未来形态

理想状态不是“游戏 SaaS”，而是“主持人驾驶舱”：

- 主持人打开页面即可继续上一轮工作
- 本地和云端自动保存，换设备也能继续
- 每一步只出现当下需要的操作
- 复制、截图、导出都足够顺手
- 成员仍然在 QQ/微信群里玩，产品只在后台托住流程

如果未来引入成员端，它也应该是轻量入口，例如提交愿望或查看自己被分配的信息，而不是完整任务系统。
